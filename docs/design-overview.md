# Design Overview

Everyone reads this first. It covers what the app is, every feature, the full tech stack, and which part of the system does what. Detailed mechanics are in the other docs.

---

## What the App Is

An accessible, AI-powered outdoor navigation app. Users describe or search for a destination in plain language, the AI understands their intent and matches it to real nearby locations. Once a route is set, the app guides them there — not with a screen full of arrows, but with 3D spatial audio (sound that genuinely feels like it's positioned in space around you) and natural-language voice guidance based on real-world landmarks, not distances.

The primary interaction mode is **voice-to-voice**: users speak naturally and the AI responds with spoken guidance. When the AI needs visual context — obstacles, signage, unfamiliar surroundings — it can request the user enable their camera for **live video streaming**. The AI sees a real-time feed, provides immediate guidance, and the user turns the camera off when done. No photo-snapping, no button-juggling.

Along the way, the AI assistant has full route context — where the user is, what waypoint they're heading toward, what the landmark hints are. It can also call backend tools (Places API, Directions API, map tiles) if it needs to cross-reference or update information mid-conversation.

The primary audience is people with visual impairments or anyone who finds screen-heavy navigation inaccessible. The audio and AI experience is designed to be genuinely better than staring at a phone screen while walking — for anyone.

## Core Features

### 1. Natural Language Destination Search (with Location Context)

User types or speaks a destination — anything from a precise address to a loose description like "the big park with the fountain downtown." The frontend sends the query and the user's current GPS coordinates to the backend. Gemini interprets the query with geographic awareness: it has trained knowledge of global geography, so coordinates like `43.6530, -79.3810` let it reason that this is downtown Toronto and weight candidates accordingly. Results are resolved to real coordinates via the Google Places API and returned as selectable cards.

### 2. Auto-Generated Walking Waypoints

The primary approach: when a session starts, the backend calls the Google Directions API in walking mode between the user's start position and the destination. The API returns a set of walking steps with coordinates and instruction text. Each step becomes a Waypoint — the instruction text becomes the raw `landmark_hint`. Gemini's narration calls then rephrase these in sensory/accessibility language before they're ever spoken. This means routes are real, accurate, and dynamically generated for any destination — not scripted.

Fallback: if Directions API integration isn't complete by the hackathon deadline, hardcoded demo routes are used. The architecture is identical — only the source of the waypoint data changes.

### 3. AI-Generated Accessible Route Overview

Before the first step, the app reads aloud a plain-English summary of the route — landmarks, textures, sounds, physical sensations. No distances or compass directions. Generated by Gemini from the waypoint list. A structured waypoint list is also shown for screen reader users.

### 4. 3D Spatial Audio (HRTF)

A continuous looping audio cue plays as the user walks. Using the Web Audio API's HRTF (Head-Related Transfer Function) engine, the sound is positioned in 3D space relative to the user — not just left/right panning, but genuinely sounding like it exists at a point in the world around them. The user turns to face the sound and it moves to centre. Walks toward it and it gets louder. All processed locally in the browser using GPS and compass data — no server round-trip for audio updates.

On waypoint arrival: loop stops, a centred arrival chime plays, next waypoint's audio begins.

### 5. Landmark-Based Accessibility Voice Narration

Default guidance mode. Instead of "turn left in 40 metres," the user hears:

> "Keep walking until you feel the pavement change texture — the kerb cut will be on your right."
> "You should hear traffic on both sides. You're at the crossing."

Generated by Gemini from the waypoint's landmark hint, spoken via the Live API voice or Web Speech API fallback. Only generated when the user crosses a meaningful distance threshold — not on every GPS tick.

### 6. Voice-to-Voice AI Assistant with Live Video

The assistant is the core interaction layer. It operates in three tiers:

**Premium (hackathon demo)**: Full voice-to-voice conversation via Gemini Multimodal Live API. The AI speaks naturally, the user speaks naturally. When the AI needs to see the environment, it asks the user to turn on the camera — a live video feed streams to Gemini in real-time. The AI gives immediate visual guidance, then tells the user they can turn the camera off. No photos, no file inputs, no interruption to conversation flow.

**Standard**: Voice-to-voice conversation with photo capture instead of live video. User takes a photo when the AI requests visual context.

**Basic**: Text chat with photo capture. Lowest token usage. AI responds in text, spoken aloud via browser TTS.

The assistant has full route context: destination, all waypoints with hints, completed progress, current target, user GPS coordinates, approximate distance. It can also call backend tools:

- `get_map_image` — fetch a Static Maps tile of the user's position for spatial reasoning
- `search_places` — re-query Places API if it needs to find something nearby
- `get_directions` — re-query Directions API for updated routing
- `get_current_location` — get the user's latest GPS fix from the session

The AI is **focused and solution-oriented**. It introduces itself briefly ("I'm NorthStar, your navigation assistant"), references conversation history intelligently ("I see you're headed to the square — looks like you may have passed the crossing"), and stays on task. It does not over-validate or chat aimlessly. It is explicitly instructed to never hallucinate — if unsure, verify via camera or say so honestly.

### 7. Agentic Function Calling

The AI assistant can call backend APIs as tools during conversation. If it needs to check nearby places, recalculate directions, fetch a map view, or get the user's latest position, it does so autonomously. Tool declarations are registered in the Live API session config. The model decides when to call them based on conversation context — no hardcoded trigger logic.

### 8. Safety & Content Moderation

**Jailbreak prevention**: Conversation history is monitored. If the AI detects deliberate attempts to circumvent its navigation purpose, it issues warnings. After 3 warnings, the session is restricted.

**Camera content moderation**: Gemini's built-in safety filters are active on all video/image input. Additionally, a custom moderation layer checks for explicit or violent content. If detected:
- Accidental exposure (user is blind, walked past something): gentle notification, no penalty.
- Deliberate intent (detected from conversation history patterns): warning issued. After 3 warnings, camera is disabled for the session.

**Connection loss handling**: If the WebSocket drops mid-conversation, the frontend plays a brief audio cue ("Connection lost"), caches the last known guidance, attempts auto-reconnect with exponential backoff, and resumes the session from where it left off. The user is never left in silence without explanation.

### 9. Client-Side Session Persistence

Sessions saved to localStorage. Reload the app and resume from where you left off. No database needed. Session includes conversation history, waypoint progress, and moderation state.

## Tech Stack

| Layer | Technology | Purpose |
|---|---|---|
| Frontend | Vanilla JS + HTML/CSS | UI, audio, maps, camera, speech |
| Backend | Python + FastAPI | Geo math, session management, AI orchestration |
| AI (text) | Gemini 2.0 Flash (`gemini-2.0-flash`) | Search, narration, function calling |
| AI (live) | Gemini 2.0 Flash Multimodal Live API | Voice-to-voice + live video streaming |
| Maps (visual) | Google Maps JS API | Map rendering — loaded directly in browser |
| Maps (search) | Google Places API | Resolve NLP search results to real coordinates |
| Maps (routing) | Google Directions API | Auto-generate walking waypoints from real routes |
| Maps (assistant) | Google Static Maps API | Fetch map tile image for AI assistant context |
| Geo math | geopy (Python) | Haversine distance, bearing |
| Audio | Web Audio API — HRTF PannerNode | 3D spatial audio — browser native |
| Voice output | Gemini Live API (premium) / Web Speech API (fallback) | AI voice responses |
| Voice input | Gemini Live API (premium) / Web Speech API (fallback) | User voice input |
| Video | MediaDevices getUserMedia | Live camera feed for AI visual context |
| Session storage | localStorage | Client-side session persistence |
| HTTP client | httpx (Python) | Backend calling external APIs |
| WebSocket | FastAPI WebSocket + google-genai Live | Bidirectional real-time AI communication |
| Env vars | python-dotenv | Keep API keys out of code |

One Google Maps API key covers all four Maps-related services. Enable Maps JS, Places, Directions, and Static Maps in Google Cloud Console.

## System Architecture

```
Browser (Vanilla JS)
  │
  ├── Google Maps JS API ────────── map rendering (direct, no backend)
  ├── Web Audio API (HRTF) ──────── 3D spatial audio (direct, no backend)
  ├── Web Speech API ────────────── TTS fallback + voice input fallback
  ├── MediaDevices getUserMedia ──── live camera video stream
  │
  └── FastAPI Backend
          │
          ├── WebSocket /live/session ─── Gemini Multimodal Live API proxy
          │     ├── Voice-to-voice (bidirectional audio)
          │     ├── Live video frames (when camera enabled)
          │     └── Function calling (tools: map, places, directions, location)
          │
          ├── REST endpoints (session mgmt, search, narration)
          │     ├── Gemini 2.0 Flash ─── search, narration, text assistant
          │     ├── Google Places API ── resolve NLP matches to coordinates
          │     ├── Google Directions API ── auto-generate walking waypoints
          │     └── Google Static Maps API ── map tile for AI assistant context
          │
          └── Moderation service ─── content safety + jailbreak detection
```

## Feature Data Flow

| Feature | Initiated | Sent to | Returns |
|---|---|---|---|
| NLP search | User types query | Backend `/search/destination` (+ user GPS) | Ranked place candidates |
| Session start | User picks place | Backend `/session/start` | Session ID, auto-generated waypoints |
| Route overview | Auto on start | Backend `/session/describe` | Plain-English route summary + structured list |
| Location update | Frontend GPS poll | Backend `/session/update` | Distance, bearing, optional narration, trigger |
| 3D audio update | Frontend (on update) | Web Audio HRTF | Sound repositioned in space |
| TTS narration | Frontend (when narration returned) | Live API voice / Web Speech API | Voice output |
| Voice assistant | User speaks | WebSocket `/live/session` → Gemini Live | Spoken AI response |
| Live video | User enables camera | WebSocket video frames → Gemini Live | Real-time visual guidance |
| Function call | AI decides mid-conversation | Backend tools (places, directions, map, location) | Data fed back to AI |
| Map image flow | Triggered by model tool call | Backend → Static Maps → Gemini | Spatial grounding response |
| Session save | Auto on state change | localStorage | Persisted JSON |
| Session resume | App boot (if saved) | Backend `/session/resume` | Restored session |
| Moderation | Continuous on all input | Backend moderation service | Warnings or restrictions |

## What Gemini Knows vs What It Looks Up

Gemini has strong geographic training data. Given GPS coordinates, it knows roughly where in the world they are — city, neighbourhood, nearby major landmarks. This is sufficient for search disambiguation (weighting candidates geographically) and for general assistant context (understanding where the user is on the route). It does not need a live API call for this.

Where it needs more: dynamic real-world conditions (construction, detours, unexpected obstacles) and precise spatial position relative to very local street features. That's what the live camera and `get_map_image` tool are for. The model decides when it needs these — it's not called on every interaction, only when the available context isn't enough to answer confidently. This keeps costs low and response times fast for straightforward questions.

## Cost & Free Tier Analysis

**Gemini 2.0 Flash free tier**: 1,500 requests/day, 1M token context window.

For hackathon testing:
- A typical navigation session: ~20–30 REST API calls (search, narration, updates) + 1 Live API session of 3–10 minutes.
- Live API sessions are billed per session, not per audio/video frame. A 10-minute voice session is roughly equivalent to a few thousand tokens of text.
- With 1,500 req/day, you can run 50+ full navigation sessions per day comfortably.
- Video frames add token load but Gemini 2.0 Flash handles this efficiently — the model processes video at reduced frame rates automatically.

**For business**: Voice-to-voice is marginally more expensive than text. Live video is the most expensive mode. The tier system lets users choose their cost/convenience tradeoff. The actual cost difference per session is small enough that premium pricing easily covers it.
